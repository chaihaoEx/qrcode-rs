# QRCode-RS v2.0 产品升级规划

## Context

当前 v1.9.0 是一个功能完整的 QR 码内容分发管理系统，具备：QR 码 CRUD、浏览器级 slot 分配、AI 评论生成、多管理员、审计日志。核心流程成熟稳定，但缺少数据洞察、外部集成、内容生命周期管理等能力。v2.0 目标是从"能用"升级到"好用"，增强运营效率和系统可扩展性。

---

## 功能分布：2 大 + 3 中 + 5 小

---

## 大功能（2 个）

### 1. 数据仪表盘 (Dashboard)

**痛点**：管理员无法直观了解系统运行状态，需要逐页翻看列表和日志才能掌握全局。

**功能设计**：
- 新增 `/dashboard` 页面作为登录后默认首页
- **统计卡片区**：
  - QR 码总数 / 今日新增
  - 今日提取次数 / 总提取次数
  - 活跃管理员数
  - 已耗尽 QR 码数量
- **图表区**：
  - 近 7 天提取趋势折线图（按天聚合，纯 CSS/SVG 实现，不引入重型图表库）
  - 近期操作动态列表（最近 10 条审计日志，实时刷新）
- **快捷入口**：创建 QR 码、查看审计日志

**技术方案**：
- 后端新增聚合查询 SQL（COUNT + GROUP BY date）
- 前端用内联 SVG 绘制简易图表，保持零 JS 依赖的风格
- 新增 `templates/dashboard.html`
- 路由：`GET /` 改为 dashboard，QR 列表移至 `GET /list`

**涉及文件**：
- `src/routes/admin.rs` — 新增 dashboard handler
- `src/routes/mod.rs` — 调整路由
- `src/services/qrcode.rs` — 新增统计查询
- `src/services/audit.rs` — 新增近期动态查询
- `templates/dashboard.html` — 新模板
- `templates/nav.html` — 新增导航项
- `static/style.css` — 仪表盘样式

---

### 2. RESTful API 接口

**痛点**：所有操作必须通过 Web UI，无法与外部系统（企业微信、自动化脚本、第三方平台）集成。

**功能设计**：
- 新增 `/api/v1/` 前缀的 JSON API
- **认证方式**：API Key（Bearer Token），存储在 `admin_users` 表新增 `api_key` 字段
- **核心接口**：
  - `POST /api/v1/qrcodes` — 创建 QR 码（JSON body）
  - `GET /api/v1/qrcodes` — 列表查询（分页 + 关键字）
  - `GET /api/v1/qrcodes/{uuid}` — 获取详情
  - `DELETE /api/v1/qrcodes/{uuid}` — 删除
  - `GET /api/v1/qrcodes/{uuid}/logs` — 提取日志
  - `GET /api/v1/stats` — 统计数据（复用 Dashboard 查询）
- **响应格式**：统一 `{"code": 0, "data": {...}, "message": "ok"}`
- API 调用同样记录审计日志

**技术方案**：
- 新增 `src/routes/api.rs` 模块
- 新增 API Key 认证中间件（不走 session，走 Authorization header）
- `admin_users` 表新增 `api_key VARCHAR(64)` 字段（migration 003）
- 用户管理页面增加 "生成 API Key" 按钮
- API 和 Web 共享 service 层，只是路由和认证不同

**涉及文件**：
- `src/routes/api.rs` — 新模块
- `src/routes/mod.rs` — 注册 API 路由
- `src/middleware.rs` — 新增 API Key 认证
- `sql/migrations/003_add_api_key.sql` — 数据库迁移
- `src/models/domain.rs` — AdminUser 新增字段
- `templates/users.html` — API Key 管理 UI

## 中功能（3 个）

### 3. QR 码有效期与归档

**痛点**：QR 码创建后永久存在，列表越来越长，无法区分活跃和历史数据。

**功能设计**：
- 创建时可设置**过期时间**（可选，默认永不过期）
  - 预设选项：1天、7天、30天、自定义日期
- 过期后：
  - 提取页显示"已过期"提示
  - 列表中灰色标记
  - 不自动删除，保留数据
- 新增**归档**功能：
  - 手动归档：将 QR 码标记为已归档
  - 归档后从默认列表隐藏，可通过筛选查看
  - 可恢复（取消归档）
- 列表页新增**状态筛选**：全部 / 活跃 / 已过期 / 已归档

**涉及文件**：
- `sql/migrations/004_add_expiry_archive.sql` — `qr_codes` 新增 `expires_at DATETIME NULL`、`is_archived TINYINT(1) DEFAULT 0`
- `src/models/domain.rs` — QrCodeRecord 新增字段
- `src/services/qrcode.rs` — 列表查询增加状态筛选
- `src/services/extract.rs` — claim_slot 检查过期
- `src/routes/admin.rs` — 创建/编辑表单增加有效期、归档按钮
- `templates/create.html`、`templates/edit.html`、`templates/list.html` — UI 调整

---

### 4. 数据导出

**痛点**：无法批量导出 QR 码数据和提取记录，运营分析依赖手动翻看。

**功能设计**：
- **QR 码列表导出**：
  - 列表页新增"导出 CSV"按钮
  - 导出字段：UUID、备注、段数、已用/最大、创建时间、最后提取时间
  - 支持当前筛选条件下导出
- **提取日志导出**：
  - 日志页新增"导出 CSV"按钮
  - 导出字段：时间、IP、Browser ID、段序号
- **审计日志导出**（超级管理员）：
  - 导出当前筛选条件下的审计记录

**技术方案**：
- 后端生成 CSV 流式响应（`Content-Type: text/csv`）
- 不引入额外依赖，手动拼接 CSV（数据量可控）
- 新增 `GET /export/qrcodes`、`GET /export/logs/{uuid}`、`GET /export/audit-logs`

**涉及文件**：
- `src/routes/admin.rs` — 新增 3 个导出 handler
- `src/routes/mod.rs` — 注册导出路由
- `templates/list.html`、`templates/logs.html`、`templates/audit_logs.html` — 添加导出按钮

---

### 5. 提取端增强

**痛点**：提取页功能单一，缺乏防滥用手段和用户引导。

**功能设计**：
- **提取频率限制**：
  - 同一 IP 每分钟最多 N 次提取请求（可配置，默认 10）
  - 超限返回 429 Too Many Requests
- **自定义提取页文案**：
  - QR 码创建时可设置提取页的标题和提示文字（可选）
  - 默认使用系统文案
- **提取成功后展示样式增强**：
  - 支持多行文本换行展示
  - 长文本自动折叠，点击展开

**涉及文件**：
- `src/rate_limit.rs` — 复用现有 RateLimiter，为提取端创建独立实例
- `src/routes/extract.rs` — 注入提取限流
- `src/config.rs` — 新增 `extract_rate_limit` 配置项
- `sql/migrations/005_add_extract_title.sql` — `qr_codes` 新增 `extract_title VARCHAR(200) NULL`
- `templates/extract.html` — 展示自定义标题、优化长文本展示
- `templates/create.html`、`templates/edit.html` — 新增提取页标题字段

## 小功能（5 个）

### 6. 深色模式

**功能**：CSS 变量切换，跟随系统 `prefers-color-scheme`，nav 增加手动切换开关，偏好存 localStorage。

**涉及文件**：`static/style.css`（新增 `@media (prefers-color-scheme: dark)` 变量覆盖）、`templates/nav.html`（切换按钮）

---

### 7. QR 码批量删除

**功能**：列表页新增复选框，选中后显示"批量删除"按钮，需要二次确认。仅支持删除，不支持批量编辑（避免误操作）。

**涉及文件**：`templates/list.html`（复选框 + 批量按钮）、`src/routes/admin.rs`（新增 `POST /batch-delete` handler）、`static/style.css`

---

### 8. 密码强度校验增强

**功能**：密码创建/修改时增加强度规则：至少包含大写、小写、数字中的两类。前端实时提示强度等级（弱/中/强），后端同步校验。

**涉及文件**：`src/services/user.rs`（`validate_password` 增加复杂度检查）、`templates/users.html`、`templates/change_password.html`（前端提示）

---

### 9. 操作确认优化

**功能**：当前删除/重置使用浏览器原生 `confirm()` 弹窗，替换为自定义模态对话框，与整体 UI 风格一致。统一封装为可复用的 confirm modal 组件。

**涉及文件**：`templates/base.html`（modal HTML 结构）、`static/style.css`（modal 样式）、`templates/list.html`、`templates/users.html`（替换 confirm 调用）

---

### 10. 审计日志自动清理

**功能**：配置文件新增 `audit_log_retention_days`（默认 90 天），服务启动时和每天凌晨自动清理过期审计日志。提取日志同理。

**涉及文件**：`src/config.rs`（新增配置项）、`src/main.rs`（启动清理任务 + tokio 定时器）、`src/services/audit.rs`（清理 SQL）

## 实施优先级建议

| 阶段 | 功能 | 理由 |
|------|------|------|
| v2.0-alpha | 6 深色模式、9 确认弹窗、8 密码强度 | 小改动，快速见效，提升体验 |
| v2.0-beta | 3 有效期与归档、5 提取端增强、10 日志清理 | 核心运营痛点，中等工作量 |
| v2.0-rc | 1 数据仪表盘、4 数据导出、7 批量删除 | 数据洞察能力，依赖前面的基础 |
| v2.0 | 2 RESTful API | 最大工作量，放最后确保核心稳定 |
